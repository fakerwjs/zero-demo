# Zero-Demo 项目优化需求清单

## 一、概述

基于代码审查报告，本清单梳理了项目中所有待修复的问题，按优先级排序，便于分步实施。

---

## 二、P0 - 紧急修复（影响核心功能）

### P0-01 Gateway 层空逻辑文件导致接口不可用

**问题描述**：`routes.go` 注册的路由使用子目录 handler，但子目录逻辑文件全是空模板；扁平化逻辑文件已实现但为死代码

**涉及文件**：
- `app/gateway/api/internal/routes.go`
- `app/gateway/api/internal/handler/product/*.go`（子目录）
- `app/gateway/api/internal/handler/order/*.go`（子目录）
- `app/gateway/api/internal/handler/payment/*.go`（子目录）
- `app/gateway/api/internal/logic/product/*.go`（子目录，空）
- `app/gateway/api/internal/logic/order/*.go`（子目录，空）
- `app/gateway/api/internal/logic/payment/*.go`（子目录，空）
- `app/gateway/api/internal/logic/*.go`（扁平化，死代码）

**优化需求**：统一 handler/logic 结构，确保所有接口有正确的逻辑实现

**优化方向**：可维护性、功能完整性

---

### P0-02 创建订单缺少用户 ID 传递

**问题描述**：`logic/createorder_logic.go` 没有从 JWT Token 中解析用户 ID 并传递给 RPC

**涉及文件**：
- [createorder_logic.go](file:///d:/work/zero-demo/app/gateway/api/internal/logic/createorder_logic.go#L38)

**优化需求**：从 JWT Token 中解析 userId，传递给 order-rpc 的 Create 方法

**优化方向**：数据完整性、安全性

---

### P0-03 创建支付缺少用户 ID 和订单号传递

**问题描述**：`logic/createpayment_logic.go` 没有传递 UserId 和 OrderNo 参数

**涉及文件**：
- [createpayment_logic.go](file:///d:/work/zero-demo/app/gateway/api/internal/logic/createpayment_logic.go#L28)

**优化需求**：从 JWT 获取 UserId，查询订单获取 OrderNo，传递给 payment-rpc

**优化方向**：数据完整性、安全性

---

### P0-04 库存扣减并发竞态条件

**问题描述**：先查询库存再扣减，高并发下可能超卖

**涉及文件**：
- [create_logic.go](file:///d:/work/zero-demo/app/order/rpc/internal/logic/create_logic.go#L33-L58)

**优化需求**：合并库存检查和扣减为原子操作，或使用分布式锁

**优化方向**：数据一致性、性能

---

### P0-05 RabbitMQ 消费者缺失

**问题描述**：通知消息发送到队列但无人消费，短信/邮件/推送功能无法实现

**涉及文件**：
- [send_logic.go](file:///d:/work/zero-demo/app/notification/rpc/internal/logic/send_logic.go#L54)
- `pkg/sender/sms.go`
- `pkg/sender/email.go`

**优化需求**：在 notification-rpc 中实现消费者，监听队列并调用发送器

**优化方向**：功能完整性

---

## 三、P1 - 重要修复（影响数据一致性和安全性）

### P1-01 订单创建失败库存未回滚

**问题描述**：先扣减库存再创建订单，订单创建失败时库存已扣减

**涉及文件**：
- [create_logic.go](file:///d:/work/zero-demo/app/order/rpc/internal/logic/create_logic.go#L55-L76)

**优化需求**：实现库存回滚机制或调整流程顺序

**优化方向**：数据一致性

---

**问题描述**：没有验证订单的用户 ID 与当前用户是否匹配，存在越权支付风险

**涉及文件**：
- [create_logic.go](file:///d:/work/zero-demo/app/payment/rpc/internal/logic/create_logic.go#L41)

**优化需求**：添加用户 ID 匹配校验

**优化方向**：安全性

## 四、P2 - 一般优化（影响性能和健壮性）

### P2-01 限流中间件内存泄漏

**问题描述**：`map[string]*rate.Limiter` 没有清理机制，持续增长

**涉及文件**：
- [ratelimitmiddleware.go](file:///d:/work/zero-demo/app/gateway/api/internal/middleware/ratelimitmiddleware.go#L16)

**优化需求**：添加定时清理机制，清理长时间未使用的限流器

**优化方向**：性能、资源管理

---

### P2-02 订单号生成重复风险

**问题描述**：使用 `UnixMilli()` 作为订单号，同一毫秒内可能重复

**涉及文件**：
- [create_logic.go](file:///d:/work/zero-demo/app/order/rpc/internal/logic/create_logic.go#L86)

**优化需求**：使用分布式唯一 ID 生成器（如雪花算法）或 UUID

**优化方向**：数据唯一性

---

### P2-03 Go 版本号不正确

**问题描述**：`go.mod` 指定 `go 1.25.5`，该版本不存在

**涉及文件**：
- [go.mod](file:///d:/work/zero-demo/go.mod#L3)

**优化需求**：修改为实际使用的版本，如 `go 1.23`

**优化方向**：环境兼容性

## 五、P3 - 建议改进（提升可观测性和开发体验）

### P3-01 缺少全局错误处理中间件

**问题描述**：错误信息直接返回给客户端，可能泄露内部实现细节

**优化需求**：添加全局错误处理中间件，统一错误响应格式

**优化方向**：安全性、用户体验

---

### P3-02 缺少请求日志中间件

**问题描述**：无法追踪请求详情，不利于问题排查

**优化需求**：添加请求日志中间件，记录请求方法、路径、耗时、状态码

**优化方向**：可观测性、运维体验

## 七、后续优化方向

### 7.1 架构层面

- **服务治理**：添加服务注册与发现、健康检查、熔断降级
- **分布式事务**：引入 Seata 或 Saga 模式解决跨服务数据一致性
- **消息队列可靠性**：添加消息确认机制、死信队列、消息重试策略

### 7.2 安全层面

- **接口签名**：对敏感操作添加请求签名验证
- **权限控制**：引入 RBAC 权限模型
- **审计日志**：完善操作审计，记录关键业务操作
- **敏感信息脱敏**：日志中敏感字段脱敏处理

### 7.3 性能层面

- **缓存策略**：添加 Redis 缓存，减少数据库压力
- **异步处理**：对非核心流程采用异步处理
- **连接池优化**：优化数据库、RPC 连接池配置

### 7.4 测试层面

- **单元测试**：完善各服务的单元测试覆盖
- **集成测试**：添加服务间集成测试
- **压力测试**：使用 JMeter 或 k6 进行性能压测

### 7.5 运维层面

- **配置中心**：引入 Apollo 或 Nacos 配置中心
- **告警机制**：基于 Prometheus 添加告警规则
- **链路追踪**：完善 OpenTelemetry 集成
