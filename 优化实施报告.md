### P0-03：创建支付时缺少用户 ID 和订单号传递

**问题描述**：创建支付时未提取用户 ID，且订单号未从订单信息中获取

**修改文件**：
- `app/gateway/api/internal/logic/payment/createpaymentlogic.go`

### P0-04：库存扣减并发竞态条件

**问题描述**：先查询库存再扣减，非原子操作，高并发下可能导致超卖

**修改文件**：
- `app/product/rpc/internal/model/product_model.go` - 修改 DeductStock 返回影响行数
- `app/product/rpc/internal/logic/deduct_stock_logic.go` - 检查影响行数判断库存是否足够
- `app/order/rpc/internal/logic/create_logic.go` - 移除提前库存检查

**修改内容**：
- SQL 更新语句增加 `WHERE stock >= amount` 条件
- 返回影响行数，为 0 时表示库存不足
- 支持负数 amount 用于库存回滚（订单取消时）

---

### P0-05：实现 RabbitMQ 消费者

**问题描述**：订单超时自动取消功能未实现，缺少延迟队列消费者

**新增文件**：
- `app/order/rpc/internal/mq/rabbitmq.go` - RabbitMQ 封装（支持延迟队列）
- `app/order/rpc/internal/consumer/order_timeout_consumer.go` - 订单超时消费者

**修改文件**：
- `app/order/rpc/etc/order.yaml` - 添加 RabbitMQ 配置
- `app/order/rpc/internal/config/config.go` - 添加 RabbitMQConf 配置结构
- `app/order/rpc/internal/svc/service_context.go` - 初始化 RabbitMQ 连接
- `app/order/rpc/order.go` - 启动消费者
- `app/order/rpc/internal/logic/create_logic.go` - 创建订单后发送延迟消息

**修改内容**：
- 使用死信交换机 + TTL 实现延迟队列
- 订单创建后发送 15 分钟延迟消息
- 消费者收到消息后检查订单状态，未支付则取消并回滚库存

### P1-01：限流中间件内存泄漏

**问题描述**：IP 限流的 buckets map 只增不减，长期运行导致内存泄漏

**修改文件**：
- `app/gateway/api/internal/middleware/ratelimitmiddleware.go`

**修改内容**：
- 添加 `limiterItem` 结构体记录最后访问时间
- 添加后台清理 goroutine，每 5 分钟清理 10 分钟未访问的 IP
- 添加最大条目限制（10000），超出时淘汰最久未访问的 IP

---

### P1-02：订单号生成重复风险

**问题描述**：仅使用时间戳生成订单号，同一毫秒内多次请求会重复

**修改文件**：
- `app/order/rpc/internal/logic/create_logic.go`

**修改内容**：
- 添加原子计数器 `orderNoCounter`
- 订单号格式改为：`ORD{timestamp}{4位序列号}`
- 使用 `atomic.AddInt64` 保证并发安全

---

### P1-03：支付创建时缺少订单信息验证

**问题描述**：支付创建时未验证用户与订单的归属关系，可能导致越权支付

**修改文件**：
- `app/payment/rpc/internal/logic/create_logic.go`

**修改内容**：
- 添加 `orderInfo.Order.UserId != in.UserId` 验证
- OrderNo 从订单信息获取，不再依赖前端传入

---

### P1-04：类型转换错误修复

## 三、后续优化方向

### 3.1 架构优化

| 方向 | 说明 | 优先级 |
|------|------|--------|
| 分布式锁 | 库存扣减使用 Redis 分布式锁替代数据库乐观锁 | 高 |
| 订单号生成器 | 使用 Redis 生成全局唯一订单号，解决多实例重复问题 | 高 |
| 消息队列解耦 | 订单创建、支付、发货等流程完全通过消息队列解耦 | 高 |
| 读写分离 | 数据库主从分离，读请求走从库 | 中 |

### 3.2 安全优化

| 方向 | 说明 | 优先级 |
|------|------|--------|
| SQL 注入防护 | 确保所有 SQL 查询使用参数化 | 高 |
| CSRF 防护 | API 接口添加 CSRF 防护 | 中 |
| 请求签名验证 | 敏感操作（支付）添加请求签名 | 高 |
| 日志脱敏 | 日志中避免打印敏感信息（密码、手机号等） | 中 |

### 3.3 性能优化

| 方向 | 说明 | 优先级 |
|------|------|--------|
| Redis 缓存 | 商品信息、订单列表等添加 Redis 缓存 | 高 |
| 接口压测 | 使用 JMeter/Gatling 进行接口压力测试 | 中 |
| 热点数据预热 | 高频访问数据提前加载到缓存 | 中 |
| 连接池调优 | 数据库连接池、RPC 连接池参数调优 | 中 |

### 3.4 运维优化

| 方向 | 说明 | 优先级 |
|------|------|--------|
| 健康检查 | 添加健康检查接口 | 高 |
| 优雅重启 | 实现服务优雅重启，无停机部署 | 中 |
| 灰度发布 | 支持灰度发布和流量切换 | 低 |
| 配置热更新 | 支持配置动态更新无需重启 | 中 |
