## 一、本次创建了什么

### 1. 目录布局（go-zero 惯用布局）

```

│   ├── api/          # 用户 HTTP API（对外，:8001）
│   └── rpc/          # 用户 gRPC（内部，:8081）

```

同时新增根级 `Makefile`、重写 `README.md`。

### 2. 用户服务（user）

- **user.rpc（gRPC，:8081，etcd key = `user.rpc`）**
  - proto：`app/user/rpc/user.proto`
  - 接口：`Register` / `Login` / `GetUserInfo`
  - 逻辑：`app/user/rpc/internal/logic/*` 已填入**桩数据**（未接数据库）
- **user-api（HTTP，:8001）**
  - 定义：`app/user/api/user.api`
  - 路由前缀 `/api/v1/user`：
    - `POST /register`（免鉴权）→ 调 user.rpc `Register`
    - `POST /login`（免鉴权）→ 调 user.rpc `Login`，并在 **API 层用 `AccessSecret` 统一签发 JWT**（claim 带 `userId`）
    - `GET /info`（JWT 鉴权）→ 从 JWT 解析 `userId`，调 user.rpc `GetUserInfo`
  - 已完成 **api → rpc 的服务发现接线**（`ServiceContext` 注入 `UserRpc` 客户端，走 etcd）

### 3. 通知服务（notification）

- **notification.rpc（gRPC，:8005，etcd key = `notification.rpc`）**
  - proto：`app/notification/rpc/notification.proto`
  - 接口：`Send` / `GetNotification` / `ListByUser`
  - 定义了 `Channel` 枚举（站内信/短信/邮件/推送）与 `NotificationInfo` 消息
  - 逻辑：`app/notification/rpc/internal/logic/*` 已填入**桩数据**（未接数据库/MQ）
  - 说明：架构图中通知服务为纯 gRPC，供其他服务内部调用，故未生成 HTTP api

## 二、踩坑记录

1. **goctl 找不到 `protoc` / 插件**：goctl 生成时会以子进程调用 `protoc`、`protoc-gen-go`、`protoc-gen-go-grpc`。本机它们分别在 `D:\protoc\bin`、`C:\Users\yihe\go\bin`，需保证这些目录在 PATH 中，否则报 `'protoc' 不是内部或外部命令`。
2. **proto 命名冲突**：通知服务里 `message Notification` 与 `service Notification` 同名，protoc 报 `already defined`。已将消息改名为 `NotificationInfo`。
