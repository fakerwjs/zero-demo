# Zero-Demo 基础设施使用文档

## 1. 概述

本项目的基础设施层包含以下核心组件：

| 组件 | 层级 | 作用 | 状态 |
|------|------|------|------|
| CloudFlare/CDN | 外层防护层 | DDoS防护、WAF、SSL管理、全球加速 | 需手动配置 |
| Nginx | 负载均衡层 | SSL卸载、限流、反向代理、健康检查 | ✅ 已配置 |
| etcd | 服务注册层 | 服务注册发现、配置存储 | ✅ 已配置 |
| etcdkeeper | 服务注册层 | etcd可视化管理界面 | ✅ 已配置 |
| ELK Stack | 日志层 | 日志采集、存储、分析、可视化 | ✅ 已配置 |
| Prometheus | 监控层 | 指标采集、告警规则 | ✅ 已配置 |
| Grafana | 监控层 | 数据可视化、仪表盘 | ✅ 已配置 |

## 2. 外层防护层（CloudFlare/CDN）

### 2.1 架构定位

```
┌─────────────────────────────────────────────────────────┐
│                    CloudFlare 防护层                    │
│  ┌─────────────────────────────────────────────────┐   │
│  │  DDoS防护  │  WAF规则  │  SSL证书  │  全球加速  │   │
│  └─────────────────────────────────────────────────┘   │
└───────────────────┬───────────────────────────────────┘
                    │ HTTPS
                    ▼
┌─────────────────────────────────────────────────────────┐
│                    Nginx 负载均衡层                     │
└─────────────────────────────────────────────────────────┘
```

### 2.2 功能说明

| 功能 | 说明 | 配置位置 |
|------|------|----------|
| DDoS 防护 | 自动防护 L3/L4/L7 层 DDoS 攻击 | CloudFlare Dashboard |
| WAF 规则 | 自定义防火墙规则，拦截恶意请求 | CloudFlare WAF |
| SSL 证书 | 免费 SSL 证书，支持 HTTPS | CloudFlare SSL/TLS |
| 全球加速 | 全球 CDN 节点，加速静态资源 | CloudFlare CDN |
| 缓存策略 | 静态资源缓存，减轻后端压力 | CloudFlare Cache |

### 2.3 使用流程

#### 步骤一：注册 CloudFlare 账号

1. 访问 https://www.cloudflare.com/
2. 注册账号并添加域名

#### 步骤二：配置 DNS

```
A 记录: zero-demo.example.com → 你的服务器 IP
CNAME 记录: www → zero-demo.example.com
```

#### 步骤三：配置 SSL/TLS

```
SSL/TLS → Edge Certificate → 开启 "Full" 模式
SSL/TLS → Origin Server → 上传服务器证书或使用 CloudFlare 证书
```

#### 步骤四：配置 WAF 规则

```
Security → WAF → Add Rule

示例规则：
1. 阻止 SQL 注入攻击
   Field: Request URI
   Operator: contains
   Value: SELECT|INSERT|UPDATE|DELETE|DROP

2. 限制请求频率
   Field: IP Address
   Operator: in
   Value: 自定义 IP 列表
   Action: Block

3. 阻止恶意 User-Agent
   Field: User-Agent
   Operator: contains
   Value: curl|python-requests|Scrapy
```

#### 步骤五：配置缓存策略

```
Caching → Configuration

缓存规则：
- Static: js, css, images (TTL: 1 hour)
- API: /api/* (TTL: 0, 不缓存)
```

### 2.4 配置示例

**CloudFlare 页面规则**：

```
规则 1: https://zero-demo.example.com/api/*
  - 缓存级别: 标准
  - 浏览器缓存 TTL: 0 秒
  - 安全级别: 高

规则 2: https://zero-demo.example.com/static/*
  - 缓存级别: 缓存一切
  - 浏览器缓存 TTL: 1 小时
  - 自动 HTTPS 重写: 开启

规则 3: https://zero-demo.example.com/*
  - 始终使用 HTTPS: 开启
  - 自动 HTTPS 重写: 开启
```

### 2.5 注意事项

- 确保服务器 IP 不直接暴露（使用 CloudFlare 代理）
- 开启 "Orange Cloud" 模式启用 CDN 代理
- 配置 "Authenticated Origin Pulls" 确保只有 CloudFlare 能访问后端
- 定期检查安全报告，更新 WAF 规则

## 3. 负载均衡层（Nginx）

### 3.1 架构定位

```
┌─────────────────────────────────────────────────────────┐
│                      Nginx 层                          │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│  │ SSL卸载  │ │ 反向代理  │ │ 健康检查  │ │ 限流保护  │  │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘  │
└───────┼───────────┼───────────┼───────────┼──────────┘
        │           │           │           │
        ▼           ▼           ▼           ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ gateway-api  │ │ user-api     │ │ 监控指标     │
│   :8000      │ │   :8001      │ │   :9100      │
└──────────────┘ └──────────────┘ └──────────────┘
```

### 3.2 配置文件

**配置位置**: `deploy/nginx/nginx.conf`

```nginx
# 核心配置结构
http {
    # 上游服务器定义
    upstream gateway {
        server host.docker.internal:8000;
    }

    upstream user-api {
        server host.docker.internal:8001;
    }

    # HTTP 转 HTTPS
    server {
        listen 80;
        return 301 https://$host$request_uri;
    }

    # HTTPS 服务
    server {
        listen 443 ssl http2;
        
        ssl_certificate /etc/nginx/ssl/cert.pem;
        ssl_certificate_key /etc/nginx/ssl/key.pem;

        location /api/v1/ {
            proxy_pass http://gateway;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }
    }
}
```

### 3.3 功能说明

| 功能 | 实现方式 | 配置位置 |
|------|----------|----------|
| SSL 卸载 | HTTPS 终止，后端使用 HTTP | `listen 443 ssl` |
| 反向代理 | 将请求转发到后端服务 | `proxy_pass` |
| 健康检查 | 检测后端服务状态 | 需额外配置 |
| 请求限流 | 限制单 IP 请求频率 | 需额外配置 |
| 静态压缩 | gzip 压缩响应 | `gzip on` |
| 请求日志 | 记录访问日志 | `access_log` |

### 3.4 使用流程

#### 步骤一：生成 SSL 证书（开发环境）

```bash
# 创建 SSL 目录
mkdir -p deploy/nginx/ssl

# 生成自签名证书（开发环境）
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout deploy/nginx/ssl/key.pem \
  -out deploy/nginx/ssl/cert.pem \
  -subj "/CN=localhost"
```

#### 步骤二：启动 Nginx

```bash
# 方式一：通过 Docker Compose 启动
cd deploy/docker
docker compose up -d nginx

# 方式二：手动启动
docker run -d --name zero-nginx \
  -p 80:80 -p 443:443 \
  -v $(pwd)/../nginx/nginx.conf:/etc/nginx/nginx.conf:ro \
  -v $(pwd)/../nginx/ssl:/etc/nginx/ssl:ro \
  nginx:1.25-alpine
```

#### 步骤三：验证配置

```bash
# 检查 Nginx 配置语法
docker exec zero-nginx nginx -t

# 查看 Nginx 日志
docker logs zero-nginx

# 测试 HTTP 转 HTTPS
curl http://localhost -v

# 测试 API 代理
curl https://localhost/api/v1/product/list -k
```

#### 步骤四：配置负载均衡（多实例）

```nginx
upstream gateway {
    server host.docker.internal:8000 weight=1;
    server host.docker.internal:8001 weight=1;
    server host.docker.internal:8002 weight=2;
    
    ip_hash;           # 会话保持
    keepalive 64;      # 连接复用
}
```

#### 步骤五：配置限流

```nginx
http {
    limit_req_zone $binary_remote_addr zone=api:10m rate=50r/s;

    server {
        location /api/v1/ {
            limit_req zone=api burst=100 nodelay;
            proxy_pass http://gateway;
        }
    }
}
```

### 3.5 Nginx 在 Docker Compose 中的配置

**配置位置**: `deploy/docker/docker-compose.yml`

## 4. 服务注册层（etcd + etcdkeeper）

### 4.1 架构定位

```

│                  服务注册发现体系                         │
│                                                         │
│  ┌──────────────┐     ┌──────────────┐                 │
│  │    etcd      │◀────│ etcdkeeper   │                 │
│  │  (:2379)     │     │   (:8080)    │                 │
│  │ 服务注册中心  │     │ 可视化管理    │                 │
│  └──────┬───────┘     └──────────────┘                 │
│         │                                              │
└─────────┼──────────────────────────────────────────────┘
          │ 服务注册 / 服务发现
          ▼
┌─────────────────────────────────────────────────────────┐
│                     微服务集群                           │
│  user-rpc | notification-rpc | product-rpc             │
│  order-rpc | payment-rpc | user-api | gateway-api      │
└─────────────────────────────────────────────────────────┘
```

### 4.2 组件说明

| 组件 | 端口 | 作用 | 配置文件 |
|------|------|------|----------|
| etcd | 2379 | 服务注册、服务发现、配置存储 | Docker Compose |
| etcdkeeper | 8080 | etcd 可视化管理界面 | Docker Compose |

### 4.3 服务注册原理

在 go-zero 框架中，服务通过配置文件自动注册到 etcd：

**服务端配置**（如 [user.yaml](file:///d:/work/zero-demo/app/user/rpc/etc/user.yaml)）：

```yaml
Name: user.rpc
ListenOn: ${USER_RPC_LISTEN:0.0.0.0:8081}
Etcd:
  Hosts:
    - ${ETCD_HOST:127.0.0.1}:${ETCD_PORT:2379}
  Key: user.rpc
```

**客户端配置**（如 [user-api.yaml](file:///d:/work/zero-demo/app/user/api/etc/user-api.yaml)）：

```yaml
UserRpc:
  Etcd:
    Hosts:
      - ${ETCD_HOST:127.0.0.1}:${ETCD_PORT:2379}
    Key: user.rpc
```

### 4.5 Docker Compose 配置

**配置位置**: `deploy/docker/docker-compose.yml`

```yaml
etcd:
  image: quay.io/coreos/etcd:v3.5.17
  container_name: zero-etcd
  command:
    - etcd
    - --name=etcd0
    - --data-dir=/etcd-data
    - --listen-client-urls=http://0.0.0.0:2379
    - --advertise-client-urls=http://127.0.0.1:2379
    - --listen-peer-urls=http://0.0.0.0:2380
  ports:
    - "2379:2379"
  volumes:
    - etcd-data:/etcd-data
  healthcheck:
    test: ["CMD", "etcdctl", "endpoint", "health"]
    interval: 10s
    timeout: 5s
    retries: 5

etcdkeeper:
  image: evildecay/etcdkeeper:latest
  container_name: zero-etcdkeeper
  ports:
    - "8080:8080"
  environment:
    - ETCD_HOST=http://etcd:2379
  depends_on:
    - etcd
```

### 4.8 常见问题

#### 4.8.1 服务无法注册到 etcd

```bash
# 检查 etcd 是否正常运行
curl http://localhost:2379/health

# 检查服务配置中的 etcd 地址是否正确
cat app/user/rpc/etc/user.yaml | grep -A5 Etcd

# 检查服务日志
cat .run/user-rpc.log | grep -i etcd
```

#### 4.8.2 etcdkeeper 无法连接 etcd

```bash
# 检查 etcdkeeper 日志
docker logs zero-etcdkeeper

# 检查网络连接
docker exec zero-etcdkeeper curl http://etcd:2379/health

# 重启 etcdkeeper
docker compose restart etcdkeeper
```

#### 4.8.3 etcd 数据持久化

etcd 数据存储在 Docker 数据卷 `etcd-data` 中，重启容器不会丢失数据。

## 5. ELK 日志平台

### 5.1 架构定位

```
┌─────────────────────────────────────────────────────────┐
│                      ELK 平台                           │
│                                                         │
│  ┌──────────┐     ┌──────────────┐     ┌──────────┐    │
│  │ Filebeat │────▶│ Elasticsearch │◀────│  Kibana  │    │
│  │ (采集)   │     │   (存储)      │     │ (可视化) │    │
│  └──────────┘     └──────────────┘     └──────────┘    │
│         │                                    │          │
└─────────┼────────────────────────────────────┼──────────┘
          │                                    │
          ▼                                    ▼
   ┌─────────────┐                    ┌─────────────┐
   │ 服务器日志  │                    │ Kibana UI   │
   │ /var/log/*  │                    │ :5601       │
   └─────────────┘                    └─────────────┘
```

### 5.2 组件说明

| 组件 | 端口 | 作用 | 配置文件 |
|------|------|------|----------|
| Filebeat | - | 日志采集器，收集服务器日志 | `deploy/filebeat/filebeat.yml` |
| Elasticsearch | 9200/9300 | 日志存储与索引 | Docker 环境变量 |
| Kibana | 5601 | 日志可视化与查询 | Docker 环境变量 |

### 5.3 使用流程

#### 步骤一：启动 ELK 服务

```bash
cd deploy/docker
docker compose up -d elasticsearch kibana filebeat
```

#### 步骤二：验证服务

```bash
# 验证 Elasticsearch
curl http://localhost:9200
# 返回：{"name":"...","cluster_name":"...","version":{"number":"8.13.0",...}}

# 验证 Kibana
curl http://localhost:5601
# 返回：Kibana 登录页面 HTML
```

#### 步骤三：配置 Filebeat

**配置文件**: `deploy/filebeat/filebeat.yml`

```yaml
filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /var/log/*.log
      - /var/log/syslog
      - /var/log/messages

output.elasticsearch:
  hosts: ["elasticsearch:9200"]

setup.kibana:
  host: "kibana:5601"

setup.dashboards.enabled: true
```

#### 步骤四：配置自定义日志采集

```yaml
filebeat.inputs:
  # 采集应用服务日志
  - type: log
    enabled: true
    paths:
      - /host/logs/*.log
    fields:
      service: zero-demo
    fields_under_root: true

  # 采集 Nginx 日志
  - type: log
    enabled: true
    paths:
      - /var/log/nginx/*.log
    fields:
      service: nginx
    fields_under_root: true
```

#### 步骤五：在 Kibana 中创建索引

1. 打开 Kibana: http://localhost:5601
2. 进入 **Management** → **Index Management**
3. 创建索引模式 `filebeat-*`
4. 选择时间字段 `@timestamp`
5. 进入 **Discover** 查看日志

### 5.4 Kibana 使用指南

#### 5.4.1 日志查询

```
# 查询所有日志
*

# 查询特定服务日志
service: "zero-demo"

# 查询错误日志
level: "error" OR message: "*error*"

# 查询时间范围
@timestamp: ["2024-01-01T00:00:00.000Z" TO "2024-01-02T00:00:00.000Z"]

# 组合查询
service: "zero-demo" AND level: "error" AND @timestamp: ["now-1h" TO "now"]
```

#### 5.4.2 创建仪表盘

1. 进入 **Dashboard** → **Create Dashboard**
2. 添加可视化组件：
   - 日志数量趋势图（Line Chart）
   - 服务分布（Pie Chart）
   - 错误率统计（Metric）
   - Top 错误日志（Table）

#### 5.4.3 设置告警

1. 进入 **Management** → **Alerting** → **Rules**
2. 创建规则：
   - 条件：错误日志数量 > 100/min
   - 动作：发送邮件/钉钉/企业微信通知

#### 6.3.5 配置告警规则

**创建告警规则文件**: `deploy/prometheus/alert_rules.yml`

```yaml
groups:
  - name: zero-demo-alerts
    rules:
      # 服务离线告警
      - alert: ServiceDown
        expr: up == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "服务 {{ $labels.job }} 已离线"
          description: "服务 {{ $labels.job }} 在 {{ $labels.instance }} 已离线超过 1 分钟"

      # 高错误率告警
      - alert: HighErrorRate
        expr: sum(http_requests_total{code=~"5.."}) / sum(http_requests_total) > 0.05
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "{{ $labels.job }} 错误率过高"
          description: "错误率: {{ $value * 100 }}%"

      # 高 CPU 使用率告警
      - alert: HighCPUUsage
        expr: (sum(rate(process_cpu_seconds_total[1m])) / count(process_cpu_seconds_total)) > 0.8
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "{{ $labels.job }} CPU 使用率过高"
          description: "CPU 使用率: {{ $value * 100 }}%"

      # 内存使用率告警
      - alert: HighMemoryUsage
        expr: process_resident_memory_bytes > 500 * 1024 * 1024
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "{{ $labels.job }} 内存使用率过高"
          description: "内存使用: {{ $value / 1024 / 1024 }} MB"
```

**更新 Prometheus 配置**:

```yaml
  - alert_rules.yml
```

## 10. 常见问题

### 10.1 Nginx 启动失败

```bash
# 检查配置语法
docker exec zero-nginx nginx -t

# 查看错误日志
docker logs zero-nginx

# 常见问题：SSL 证书不存在
# 解决方案：生成自签名证书
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout deploy/nginx/ssl/key.pem \
  -out deploy/nginx/ssl/cert.pem \
  -subj "/CN=localhost"
```

### 10.2 Prometheus 无法抓取指标

```bash
# 检查目标状态
curl http://localhost:9090/api/v1/targets

# 检查服务是否启动
curl http://localhost:9100/metrics

# 常见问题：host.docker.internal 无法解析
# 解决方案：使用实际 IP 地址
```

### 10.3 Elasticsearch 启动失败

```bash
# 检查内存配置
docker logs zero-elasticsearch

# 常见问题：内存不足
# 解决方案：增加 ES_JAVA_OPTS
# docker-compose.yml 中设置 ES_JAVA_OPTS=-Xms256m -Xmx256m
```

### 10.4 Grafana 无法连接 Prometheus

```bash
# 检查数据源配置
curl -u admin:admin http://localhost:3000/api/datasources

# 测试连接
curl http://prometheus:9090
```

### 10.5 Filebeat 无法采集日志

```bash
# 测试输出
docker exec zero-filebeat filebeat test output

# 检查日志
docker logs zero-filebeat
```

## 11. 生产环境建议

### 11.1 Nginx

- 使用正式 SSL 证书（Let's Encrypt）
- 配置多实例负载均衡
- 启用健康检查
- 配置请求限流
- 开启访问日志并接入 ELK

### 11.2 ELK

- Elasticsearch 使用集群模式（至少 3 节点）
- 配置索引生命周期管理（ILM）
- 设置日志轮转和清理策略
- 启用 Elasticsearch 安全认证

### 11.3 Prometheus

- 使用远程存储（如 Thanos）
- 配置告警规则和通知渠道
- 设置服务发现（Consul/Etcd）
- 启用 Prometheus 高可用

### 11.4 Grafana

- 配置 LDAP/SSO 认证
- 设置仪表盘权限控制
- 启用告警通知（邮件/钉钉/企业微信）
- 定期导出仪表盘配置

### 11.5 CloudFlare

- 启用 Authenticated Origin Pulls
- 配置 WAF 规则和速率限制
- 开启 Always Use HTTPS
- 设置页面规则缓存策略
